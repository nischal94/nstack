nstack is the **AI-native quality layer** for founders building to scale:
1. **Complement, don't compete** — never duplicates superpowers' dev loop; hands off at natural boundaries.
2. **Zero mandatory setup for core skills** — Markdown-only for core; design cluster is the one opt-in exception (Bun + Playwright via `./setup`).

5. **AI-native as primary lens** — LLM/prompt/model-boundary risk is first-class, not an afterthought.

A gap that fails any filter is not a gap — it's a non-goal.

## Phase 2 — Overlap matrix (nstack ↔ superpowers)

**Verdict: nstack is clean of superpowers duplication.** Every point of apparent overlap has a crisp, intentional boundary.

| nstack skill | superpowers skill that could overlap | Actual boundary |
|---|---|---|
| `/review` | `requesting-code-review`, `receiving-code-review` | `/review` is inline **execution** of a staff-engineer review on the current diff (auto-fix + flag). superpowers skills are **meta-discipline** — how to ask for review, how to receive feedback. Different jobs. |
| `/autoplan` | `writing-plans` | `/autoplan` **reviews** an already-written plan (scope, architecture, test matrix). `writing-plans` **authors** plans from a spec. Explicit handoff exists. |
| `/investigate` | `systematic-debugging` | `/investigate` does **triage** (timeline, suspect range, hypothesis with confidence). `systematic-debugging` **executes** root-cause fix. `/investigate` explicitly hands off. |
| `/premise` | `brainstorming` | `/premise` challenges **whether to build** (kill bad ideas). `brainstorming` explores **what to build** (after deciding to). Adjacent, not overlapping. |
| `/office-hours` | `brainstorming` | `/office-hours` is **structured YC-style forcing questions** (demand reality, wedge, observation). `brainstorming` is open exploration of intent. Different protocol. |
| `/council` | `brainstorming` | `/council` is **multi-persona adversarial deliberation** on a committed decision. `brainstorming` is exploratory. Different shape. |
| `/ship` | `finishing-a-development-branch`, `requesting-code-review` | `/ship` is the **pipelined execution** (tests → review → version → PR). superpowers covers the **decision framework** around branch completion. `/ship` explicitly delegates review to superpowers. |

**Key insight:** superpowers owns **process and discipline** (the *how* of planning, debugging, reviewing, verifying). nstack owns **artifacts and execution** (the security audit report, the migration dry-run, the design mockup, the health score). The two packs are philosophically complementary by design, not by accident.

## Phase 4 — Frontier gaps (2026 AI-native coverage neither pack has)

These are the skills that would actually move nstack's thesis forward. All four pass the zero-setup filter (pure Markdown; no new runtime requirement).

### 🟢 HIGH: `/mcp-audit` — MCP server supply chain

**Gap:** `/cso`'s Phase 8a already covers *skill* supply chain (prompt injection in SKILL.md, overly broad `allowed-tools`). But **MCP servers** are a separate and larger 2026 attack surface. Users install MCP servers from npm/GitHub that get broad file/network/shell access through an LLM that can be prompted by adversarial content.

**What it does:**
- Scans `.mcp.json`, `~/.claude/*.json`, and active MCP config for installed servers
- Per server: network access scope, file access scope, whether it runs remote code (`npx`, GitHub URLs), whether it pipes through untrusted content to tool descriptions
- Flags unvetted servers, over-scoped permissions, stdio vs HTTP exposure
- Checks for prompt injection in tool descriptions (same pattern as skill supply chain)

**Thesis fit:** ✅ AI-native (MCP is an AI-era primitive), ✅ zero setup (grep + JSON parsing), ✅ founder-first (every founder installs MCPs without reading them), ✅ superpowers-compatible (unique scope), ✅ zero noise (each finding = concrete capability + concrete exploit).

**Priority reasoning:** MCP is the most-installed, least-audited AI attack surface in 2026. nstack's thesis *demands* this.

---

### 🟢 HIGH: `/rag-audit` — RAG pipeline safety

**Gap:** Neither pack covers retrieval-augmented generation pipelines. RAG is everywhere in 2026 and has its own attack surface: data poisoning, retrieval injection, chunk boundary leaks, hallucinated citations, PII leaking from source docs into prompts.

**What it does:**
- Detects RAG usage (vector store imports, embedding calls, retrieval patterns)
- Per pipeline: source validation (where does the corpus come from), chunk/embedding pipeline hygiene, retrieval-injection detection (user input influencing retrieval ranking), citation verification (model claims matching retrieved text), PII scan on source docs
- Flags unvalidated source corpora, missing citation enforcement, no retrieval confidence gates

**Thesis fit:** ✅ AI-native (RAG is peak AI-native), ✅ zero setup (static code analysis), ✅ each finding = exploit path, ✅ complements `/cso` as a focused vertical.

---

### 🟢 HIGH: `/cost-audit` — LLM cost governance

**Gap:** `/cso` flags missing `max_tokens` as a financial-DoS issue, but there's no skill that looks at **architectural cost posture**: caching, batching, fallback-model selection, per-user rate limits, token counting before send, context-window waste, prompt caching usage.

**What it does:**
- Scans every LLM call site in the codebase
- Per call site: max_tokens set, model cost tier, cacheable (Anthropic prompt caching / OpenAI cache control), inside user-triggered loop (O(n) spend risk), per-user rate limit upstream, fallback tier (cheaper model on degraded service)
- Architectural findings: unbounded background jobs, missing caching on stable system prompts, O(n) per-message calls, no token pre-flight, expensive model used where cheap would do
- Outputs projected cost per active user at N queries/day

**Thesis fit:** ✅ AI-native (cost IS the AI-native failure mode after security), ✅ zero setup (grep + AST), ✅ founder-first (unbounded spend kills founders overnight), ✅ each finding = measurable $ impact.

**Priority reasoning:** "Unbounded LLM spend as financial DoS" is explicitly named in ETHOS.md as a gap traditional tools miss. This skill closes that loop.

---

### 🟡 MEDIUM: `/agent-safety` — tool-use blast radius

**Gap:** As more products ship agent loops (the model calls tools in a loop), the blast radius of an LLM-chosen tool call becomes a first-class safety concern. Neither pack covers: tool permission scoping, dry-run/approval flows for destructive tools, sandboxing, tool-call rate limits, loop-detection.

**What it does:**
- Identifies agent-loop architectures in the code
- Per agent: tool inventory, per-tool blast radius (filesystem write, shell, HTTP, DB), approval/dry-run gates present, rate limits, loop-break heuristics
- Flags destructive tools without user approval, agent loops without iteration caps, tool descriptions that can be prompt-injected

**Thesis fit:** ✅ AI-native, ✅ zero setup.

**Priority note:** Might fold cleanly into `/cso` as a new phase (like Phase 8a did for skill supply chain) rather than a standalone skill. Decision can wait.
